# tldr スキル

既に書かれた長い文章を「先に要点、詳細は補足として後ろに置く」構成に再整形する。**新規に文章を書き起こすスキルではなく、既存の文章を整形し直すスキル**である点に注意する。

## 対象

- 引数でファイルパス / URL が指定された場合はそれを対象にする
- 引数が無い場合は、直近の会話で自分が出力した長い文章（調査結果・提案・レポート等）を対象にする
- 対象が不明瞭な場合は `AskUserQuestion` で確認する

## 適用するかどうかの判断

このスキルは「読む人が要点だけ先に知りたい、詳細は根拠として後から参照したい」文章にのみ適用する。以下のような場合は無理に2部構成にせず、その旨を報告して終了する。

- 元の文章が既に数行程度で短く、分割する意味がない
- 単純な事実確認・一問一答的な回答で、メカニズムやトレードオフを含まない
- コード差分そのもの、単純なコマンド出力など、要約すると情報が失われる種類のもの

## 実行フロー

### Step 1: 元の文章を読む

対象ファイル/出力を `Read` するか、会話コンテキストから該当箇所を特定する。

### Step 2: 「要点」に上げるべき情報を抽出する

以下の基準で、意思決定に直結する情報だけを抜き出す。

- **どうやって達成しているか / 結論は何か**（メカニズム・結果を1〜数行で）
- **トレードオフ・リスク・注意点**（同じ方法を採用した場合に何が起きるか、良い点と注意点を対にする）
- 30秒〜1分程度で読み切れる分量に収める。箇条書き・表を優先し、長い説明文にしない
- 元の文章に無い推測を新たに付け加えない。あくまで元の文章の中から「意思決定に効く部分」を選び出して先頭に持ってくる作業

### Step 3: 「詳細」に残す情報を整理する

要点に上げなかった残りの情報（処理フロー、該当コード、手順、技術的根拠など)を「詳細（補足情報）」として要点セクションの後ろに配置する。

- 元の文章の情報を削除しない。詳細セクションは「要点の根拠」として機能させる
- 詳細セクション内は元の見出し構成をできるだけ活かし、無理に書き直さない

### Step 4: 出力構成

以下の見出し構成で組み立てる（既存の背景・ゴール等のセクションがあれば、要点より前に維持してよい）。

```markdown
## 要点（サマリー）

### どうやって達成しているか / 結論

- ...

### トレードオフ・リスク

| 観点 | 内容 |
| -- | -- |
| ✅ 良い点 | ... |
| ⚠️ 注意点 | ... |

---

## 詳細（補足情報）

（元の文章の技術的根拠・コード・手順をそのまま配置）
```

### Step 5: 反映方法の確認

- 対象がファイル / 外部サービス（Linear Issue 本文など）の場合、上書き前に変更内容を提示し、反映してよいか確認してから `Edit` や該当ツールで反映する
- 対象が会話出力のみの場合は、整形結果をそのままチャットに出力する
